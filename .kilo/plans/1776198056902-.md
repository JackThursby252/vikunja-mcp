# Fix Vikunja MCP API Compatibility Issues

## Context

The issue ([aimbitgmbh/vikunja-mcp#1](https://github.com/aimbitgmbh/vikunja-mcp/issues/1)) reports that the `tasks_list_all` tool breaks on Vikunja 1.0.0+ due to a renamed endpoint. The `/tasks/all` → `/tasks` fix was already applied. This plan covers **additional** API incompatibilities found by comparing every endpoint in `src/client.ts` against the current Vikunja API Swagger spec (v2.3.0).

## Findings

### 1. ALREADY FIXED: `/tasks/all` → `/tasks`
- `src/client.ts:138` and `src/client.ts:436` — already changed to `/tasks`.

### 2. `updateLabel` uses wrong HTTP method
- **File:** `src/client.ts:251`
- **Current:** `POST /labels/{id}`
- **API expects:** `PUT /labels/{id}`
- **Impact:** Label updates will fail with a 405 or unexpected behavior.

### 3. `createSubscription` uses wrong HTTP method
- **File:** `src/client.ts:484`
- **Current:** `POST /subscriptions/{entity}/{entityId}`
- **API expects:** `PUT /subscriptions/{entity}/{entityId}`
- **Impact:** Creating subscriptions will fail.

### 4. `getSubscription` calls a non-existent endpoint
- **File:** `src/client.ts:473`
- **Current:** `GET /subscriptions/{entity}/{entityId}`
- **API reality:** No GET endpoint exists for subscriptions. Subscription status is included in the task/project response payload (as a `subscription` field) when fetching single items.
- **Impact:** Checking subscription status will fail with 404/405.
- **Fix:** Fetch the entity (task or project) directly and read the `subscription` field from the response, OR simply try/catch the existing approach if the 404 is graceful (current code already does `try/catch` returning `null`).

### 5. `getNotification` calls a non-existent endpoint
- **File:** `src/client.ts:454`
- **Current:** `GET /notifications/{id}`
- **API reality:** `/notifications/{id}` only supports `POST` (mark as read/unread), not `GET`.
- **Impact:** Getting a single notification by ID will fail.
- **Fix:** Remove dedicated `getNotification` method; instead filter from the full list via `GET /notifications`, or just remove the `notification_get` tool.

### 6. `deleteNotification` calls a non-existent endpoint
- **File:** `src/client.ts:461`
- **Current:** `DELETE /notifications/{id}`
- **API reality:** `/notifications/{id}` only supports `POST` (mark as read/unread).
- **Impact:** Deleting/dismissing notifications will fail.
- **Fix:** Change to `POST /notifications/{id}` to mark notification as read. The tool semantics shift from "delete" to "mark as read" since the API doesn't support actual deletion — rename tool to `notification_mark_read` or keep "dismiss" semantics with POST underneath.
